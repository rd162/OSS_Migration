    48→
    49→### Why critique sees all 3
    50→
    51→Cross-solution assessment requires comparative context.
    52→A critique agent seeing only one candidate cannot identify
    53→that candidate B already solved a weakness present in A,
    54→or that all three share a common blind spot.
    55→
    56→### Why solution authors isolate
    57→
    58→Independent improvement prevents convergence toward a single design.
    59→If Author A sees critique of B and C,
    60→it anchors toward avoiding their weaknesses

    62→
    63→### Why no role-game framing
    64→

   224→### Research-Armed Condorcet (Standard+ depth)
   225→
   226→Condorcet voters research key claims before selecting a winner.

The PHP source returns loosely-typed JSON with strings where integers might be expected, null where empty strings might appear, and HTML fragments as string values. Achieving byte-level JSON parity through Pydantic's serialization layer requires extensive `json_encoders` customization and careful handling of None vs empty string vs missing key. Solution C does not acknowledge this impedance mismatch.

**R4 (Plugin/extension compatibility) -- COMPLICATED BY ASYNC.** The plugin system (24 hooks) must be async-aware if the framework is async. Hooks like `HOOK_ARTICLE_FILTER` and `HOOK_FEED_FETCHED` fire during feed processing. FastAPI's routing model assumes path-based REST endpoints. **R10 (Each phase independently testable) -- VIOLATED BY PASS COUNT.** 11 passes means 11 integration boundaries to test. Passes 1-6 produce no runnable HTTP endpoint.      8→## Context
     9→
    10→The PHP codebase stores user passwords in the `ttrss_users` table (`pwd_hash` column) using several hash formats:
    11→
    12→- **Unsalted SHA1**: `SHA1:<sha1_hash>` (e.g., `SHA1:5baa61e4...`)
    13→- **Salted SHA1**: `SHA1X:<salt>:<sha1_hash>`
    14→- **Salted SHA256 (MODE2)**: `MODE2:<sha256_hash>` (salt prepended to password before hashing)
    15→
    16→SHA1 is cryptographically broken for collision resistance and considered inadequate for password storage — it lacks key stretching, making brute-force attacks feasible with modern GPUs.
    17→
    18→The Python migration must upgrade to a modern password hashing algorithm (bcrypt, argon2, or scrypt) while preserving the ability for existing users to log in without a forced password reset.
    19→
    20→## Options
    21→
    22→### A: Dual-Hash with Gradual Migration
    23→
    24→On login, check the hash format:
    25→1. If legacy (`SHA1:...`, `SHA1X:...`, or `MODE2:...`): verify with the appropriate legacy algorithm, then re-hash with bcrypt/argon2 and update the DB row
    26→2. If modern (`$2b$...` or `$argon2id$...`): verify with the modern algorithm directly
    27→
    28→Over time, all active users migrate automatically. Dormant accounts retain old hashes until login or admin action.
    29→
    30→### B: Big-Bang Migration (Force Reset All)
    31→
    32→Invalidate all existing password hashes. Send password reset emails to all users. New passwords are hashed with bcrypt/argon2 from day one.
    33→
    34→- Clean break — no legacy hash code
    35→- Disruptive to users (especially if email is not configured)
    36→- Dormant accounts handled cleanly
    37→
    38→### C: Wrap-Hash Migration
    39→
    40→Hash existing SHA1 hashes with bcrypt: `bcrypt(sha1(password))`. Store with a flag indicating the double-hash. On login, compute `sha1(input)` then `bcrypt.verify()`. Optionally re-hash to pure bcrypt on successful login.
    41→
    42→- No user disruption
    43→- All hashes upgraded in one DB migration
    44→- Slightly more complex verification logic
    45→- SHA1 weakness is mitigated by outer bcrypt layer
    46→
    47→## Trade-off Analysis
    48→
    49→| Criterion | A: Dual-Hash Gradual | B: Big-Bang Reset | C: Wrap-Hash |
    50→|-----------|---------------------|-------------------|--------------|
    51→| User disruption | None | High (all users reset) | None |
    52→| Implementation complexity | Medium | Low | Medium |
    53→| Legacy code retention | Temporary (login check) | None | Permanent (double-hash) |
    54→| Dormant account handling | Remains SHA1 until login | Forced reset | All upgraded immediately |
    55→| Security posture during migration | Gradual improvement | Immediate full security | Immediate (bcrypt wraps SHA1) |
    56→| Password reset email dependency | No | Yes (critical) | No |
    57→| Rollback complexity | Low | High (hashes destroyed) | Low |
    58→

    60→

    62→
    63→Target hash algorithm: **argon2id** (via the `argon2-cffi` library) as the primary choice, with bcrypt as fallback if argon2 is unavailable.
    64→
    65→## Decision
    66→

    10→The PHP codebase encrypts feed authentication credentials (HTTP Basic Auth passwords stored in `ttrss_feeds.auth_pass`) using the `mcrypt` extension with AES-128-CBC.     11→
    12→The Python migration must:
    13→- Decrypt existing feed passwords encrypted with mcrypt AES-128-CBC
    14→- Re-encrypt them with a modern, maintained encryption library
    15→- Support key rotation in the future
    16→- Handle the transition transparently (feeds must continue to authenticate)
    17→
    18→## Options
    19→
    20→### A: Fernet (cryptography Library)
    21→
    22→Use `Fernet` from the `cryptography` package. Fernet provides AES-128-CBC with HMAC-SHA256 authentication, key derivation, and timestamped tokens. It is a high-level, misuse-resistant API.
    23→
    24→- Built-in authentication (HMAC) prevents tampering
    25→- Timestamped tokens enable key rotation policies
    26→- Single-key symmetric encryption
    27→- Well-audited `cryptography` library
    28→
    29→### B: AES-GCM (cryptography Library)
    30→
    31→Use `AES-GCM` directly from the `cryptography` package. GCM provides authenticated encryption with associated data (AEAD). More control than Fernet but more room for misuse.
    32→
    33→- Authenticated encryption (built-in integrity)
    34→- Requires manual nonce management (critical to never reuse)
    35→- Industry standard for modern encryption
    36→- Lower-level API than Fernet
    37→
    38→### C: Keep mcrypt-Compatible Format
    39→
    40→Use a Python mcrypt-compatible library (e.g., `pycryptodome` in mcrypt compat mode) to read/write the same AES-128-CBC format. This allows the PHP and Python systems to share the same encrypted values during a transition period.
    41→
    42→- Zero migration effort for existing data
    43→- Maintains deprecated, unauthenticated encryption (CBC without HMAC)
    44→- Vulnerable to padding oracle attacks
    45→- No path forward for key rotation
    46→

    48→
    49→| Criterion | A: Fernet | B: AES-GCM | C: mcrypt-Compatible |
    50→|-----------|-----------|------------|---------------------|
    51→| Security (authenticated encryption) | Yes (HMAC-SHA256) | Yes (GCM tag) | No (CBC only) |
    52→| Misuse resistance | High (opinionated API) | Medium (nonce management) | Low |
    53→| Migration script needed | Yes (decrypt old, re-encrypt) | Yes (decrypt old, re-encrypt) | No |
    54→| Key rotation support | Built-in (MultiFernet) | Manual | None |
    55→| Library maintenance | Excellent (pyca/cryptography) | Excellent (pyca/cryptography) | Poor (mcrypt deprecated) |
    56→| PHP interoperability during transition | No (different format) | No (different format) | Yes |
    57→| Implementation complexity | Low | Medium | Low |
    58→| Padding oracle resistance | N/A (authenticated) | N/A (authenticated) | Vulnerable |
    59→
    60→## Preliminary Recommendation
    61→
    62→**Option A (Fernet)** — provides the strongest security guarantees with the simplest API. The migration path is:
    63→
    64→1. Write a one-time migration script that reads existing mcrypt-encrypted passwords using `pycryptodome` (AES-128-CBC decrypt with the existing `FEED_CRYPT_KEY`)

    67→4. Mark migrated rows (e.g., prefix `fernet:` or a separate column)

    70→
    71→## Decision
    72→

    83→
