The production config has `RATELIMIT_ENABLED=True`. `config.setdefault` finds the key present and returns `True`. `self.enabled = True`. The shared `limiter` singleton now has `enabled=True`. If any further `create_app()` is called with `RATELIMIT_ENABLED=False`, the same `setdefault` will find the key already set to `False` and `self.enabled=False`. The order of test execution determines whether rate limiting is enabled or disabled in shared state. This is not the direct cause of the 302 failures but is a correctness hazard.

## FLAW 4 — Five Other Module-Level Extension Singletons Are All Subject to the Same Multi-App Init Problem

`db`, `login_manager`, `sess`, `csrf`, and `limiter` are all module-level singletons. Flask's `init_app` pattern is designed for a single app per process or for apps that do not share extension singletons. When multiple `create_app()` calls occur in the same process (which happens during testing), every `init_app()` call can mutate shared singleton fields. `flask_wtf.CSRFProtect` has `_exempt_views` and `_exempt_blueprints` as instance-level sets shared across all apps. `flask_session.Session.init_app` is safe (only writes to `app.session_interface`) but `flask_login.LoginManager.init_app` registers `after_request` hooks on every app it's called with, accumulating hooks on throwaway apps and leaking `after_request` registrations.

## FLAW 5 — `test_tasks_digest.py` Tests Call `send_headlines_digests(app=None)` Without Preventing the Internal `create_app()` Call

The unit tests in `test_tasks_digest.py` patch `ttrss.extensions.db` and `ttrss.prefs.ops.get_user_pref`, but do not patch `ttrss.create_app` or `ttrss.tasks.digest.create_app`. The fallback branch in `send_headlines_digests` that calls `create_app()` when `app=None` is not patched out. As a result, a full production app factory is invoked during unit test execution, causing all module-level extension singletons to be re-initialized with production configuration.

## FLAW 6 — `send_headlines_digests` Should Not Call `create_app()` Internally

`send_headlines_digests` is a business logic function. Business logic functions must not invoke application factories. The application context is the caller's responsibility. A Celery task or CLI script that invokes this function must push an app context before calling it. The internal `create_app()` fallback is architecturally incorrect: it couples domain logic to infrastructure setup, it uses production config silently, and it mutates shared extension singletons as a side effect.

## FLAW 7 — Session-Scoped `app` Fixture Is Contaminated by Unit Tests That Use It

Placing them in `tests/unit/` alongside pure unit tests that define their own minimal Flask apps creates confusion about fixture scope and execution dependencies.

## FLAW 8 — `tests/unit/` Contains Tests That Are Not Unit Tests

`test_plugin_storage.py`, `test_plugins_storage.py`, `test_rate_limiting.py`, `test_errors.py`, `test_prefs_blueprint.py`, `test_loader.py`, `test_plugins_loader.py` all use `app` and/or `client` fixtures backed by real Postgres and Redis. ## FLAW 9 — `test_tasks_digest.py` Contains No Test For the `app=None` Fallback Path That Actually Guards It

The test file covers `send_headlines_digests` but never patches the internal `create_app()` call. The tests accidentally exercise the `create_app()` fallback as a side effect of passing `app=None`. ## FLAW 10 — Flask-Talisman Is Not Safe for Multi-App Singleton Usage and This Is a Documented Limitation

Flask-Talisman's `init_app()` method stores `self.force_https`, `self.app`, and all other security policy settings as singleton instance attributes. This is unlike Flask-Login, Flask-WTF, Flask-SQLAlchemy, Flask-Session, and Flask-Limiter, which store per-app state in `app.extensions`. Flask-Talisman's design means it inherently cannot be used as a module-level singleton in a factory pattern that creates multiple apps. The `__init__.py` comment "init_app pattern avoids circular imports (AR02)" is incorrect as justification for using it as a singleton — this concern applies only to `db`, which actually uses Flask's `app.extensions` correctly.

## FLAW 11 — `create_app()` Calls `talisman.init_app(app, force_https=not app.testing)` Which Does Not Persist Correctly

The `force_https=not app.testing` argument is evaluated at call time and written to `talisman.force_https`. On subsequent `create_app()` calls — even with non-testing apps — this value is overwritten. The current architecture makes `force_https` a race condition between `create_app()` invocations.
