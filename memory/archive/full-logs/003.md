     1→# ADR-0008: Password Hash Migration
     2→
     3→- **Status**: proposed
     4→- **Date proposed**: 2026-04-03
     5→- **Deciders**: TBD
     6→
     7→## Context
     8→
     9→The PHP codebase stores user passwords in the `ttrss_users` table (`pwd_hash` column) using several hash formats:
    10→
    11→- **Unsalted SHA1**: `SHA1:<sha1_hash>` (e.g., `SHA1:5baa61e4...`)
    12→- **Salted SHA1**: `SHA1X:<salt>:<sha1_hash>`
    13→- **Salted SHA256 (MODE2)**: `MODE2:<sha256_hash>` (salt prepended to password before hashing)
    14→
    15→SHA1 is cryptographically broken for collision resistance and considered inadequate for password storage — it lacks key stretching, making brute-force attacks feasible with modern GPUs.
    16→
    17→The Python migration must upgrade to a modern password hashing algorithm (bcrypt, argon2, or scrypt) while preserving the ability for existing users to log in without a forced password reset.
    18→
    19→## Options
    20→
    21→### A: Dual-Hash with Gradual Migration
    22→
    23→On login, check the hash format:
    24→1. If legacy (`SHA1:...`, `SHA1X:...`, or `MODE2:...`): verify with the appropriate legacy algorithm, then re-hash with bcrypt/argon2 and update the DB row
    25→2. If modern (`$2b$...` or `$argon2id$...`): verify with the modern algorithm directly
    26→
    27→Over time, all active users migrate automatically. Dormant accounts retain old hashes until login or admin action.
    28→
    29→### B: Big-Bang Migration (Force Reset All)
    30→
    31→Invalidate all existing password hashes. Send password reset emails to all users. New passwords are hashed with bcrypt/argon2 from day one.
    32→
    33→- Clean break — no legacy hash code
    34→- Disruptive to users (especially if email is not configured)
    35→- Dormant accounts handled cleanly
    36→
    37→### C: Wrap-Hash Migration
    38→
    39→Hash existing SHA1 hashes with bcrypt: `bcrypt(sha1(password))`. Store with a flag indicating the double-hash. On login, compute `sha1(input)` then `bcrypt.verify()`. Optionally re-hash to pure bcrypt on successful login.
    40→
    41→- No user disruption
    42→- All hashes upgraded in one DB migration
    43→- Slightly more complex verification logic
    44→- SHA1 weakness is mitigated by outer bcrypt layer
    45→
    46→## Trade-off Analysis
    47→
    48→| Criterion | A: Dual-Hash Gradual | B: Big-Bang Reset | C: Wrap-Hash |
    49→|-----------|---------------------|-------------------|--------------|
    50→| User disruption | None | High (all users reset) | None |
    51→| Implementation complexity | Medium | Low | Medium |
    52→| Legacy code retention | Temporary (login check) | None | Permanent (double-hash) |
    53→| Dormant account handling | Remains SHA1 until login | Forced reset | All upgraded immediately |
    54→| Security posture during migration | Gradual improvement | Immediate full security | Immediate (bcrypt wraps SHA1) |
    55→| Password reset email dependency | No | Yes (critical) | No |
    56→| Rollback complexity | Low | High (hashes destroyed) | Low |
    57→
    58→## Preliminary Recommendation
    59→
    60→**Option A (Dual-Hash with Gradual Migration)** — least disruptive, straightforward to implement, and the legacy SHA1 verification code is small and isolated. Active users migrate transparently on their next login. For dormant accounts, add an admin tool or scheduled job to flag accounts that have not migrated after N months, optionally forcing a reset for those users only.
    61→
    62→Target hash algorithm: **argon2id** (via the `argon2-cffi` library) as the primary choice, with bcrypt as fallback if argon2 is unavailable.
    63→

    65→

    72→- If Option A: dormant accounts remain on SHA1 indefinitely unless forced

    33→- Requires manual nonce management (critical to never reuse)
    34→- Industry standard for modern encryption
    35→- Lower-level API than Fernet

    38→

    42→- Maintains deprecated, unauthenticated encryption (CBC without HMAC)

    44→- No path forward for key rotation

   157→## Migration Flow Variants
   158→
   159→### Variant A: Entity-First (Bottom-Up)
   160→
   161→**Strategy**: Migrate database models first, then build business logic on top.
   162→
   163→```
   164→Pass 1: SQLAlchemy models + Alembic migrations (all 35 tables)
   165→Pass 2: Database access layer (repository pattern or direct ORM)
   166→Pass 3: Core utilities (config, auth, session, preferences)
   167→Pass 4: Feed engine (parser, updater, cache)
   168→Pass 5: API handlers (RPC, Feeds, Article, Pref_*)
   169→Pass 6: Plugin system
   170→Pass 7: Frontend adaptation (or keep existing)
   171→Pass 8: Deployment (Docker, CI/CD)
   172→```
   173→
   174→**Pros**:
   175→- Clean foundation — models drive everything
   176→- Can validate schema early with real data
   177→- Natural for SQLAlchemy-centric development
   178→
   179→**Cons**:
   180→- Long time before any runnable code
   181→- Hard to test without handlers
   182→- Models may need revision when business logic reveals patterns
   183→
   184→### Variant B: Call-Graph-First (Top-Down)
   185→
   186→**Strategy**: Migrate along the call graph, starting from entry points and working inward.
   187→
   188→```
   189→Pass 1: Entry points + bootstrap (Flask app, config, routing)
   190→Pass 2: Handler base classes + dispatch logic
   191→Pass 3: Core handlers (Feeds, Article, RPC) with stub DB
   192→Pass 4: Database layer (replace stubs with real ORM)
   193→Pass 5: Feed engine
   194→Pass 6: Preference handlers
   195→Pass 7: Plugin system
   196→Pass 8: Polish + deployment
   197→```
   198→
   199→**Pros**:
   200→- Runnable skeleton early
   201→- Natural for agile/incremental development
   202→- Can use stubs to test routing before DB is ready
   203→
   204→**Cons**:
   205→- Stub management overhead
   206→- May need to refactor handlers when DB layer solidifies
   207→- Risk of building on wrong abstractions
   208→
   209→### Variant C: Vertical Slice (Feature-First)
   210→
   211→**Strategy**: Migrate complete vertical slices — one feature at a time, including its models, logic, API, and frontend.
   212→
   213→```

   217→Slice 4: Feed Update Engine (parser + daemon + cache)

   220→Slice 7: Preferences + Profiles
